# CLAUDE.md — flowx

`flowcast` 是一个轻量 workflow 编排框架：断点续跑、HITL、多 CLI/agent 调度、
自改安全沙箱、质量门，以及在其之上的 L3 codegen 编排层（动态生成并执行 flow）。
**零运行时依赖、纯 ESM、Node ≥ 20。**

> **新 session 先读 [docs/BACKGROUND.md](docs/BACKGROUND.md)**——flowx 的来龙去脉与愿景
> （统一自我迭代引擎、三层架构、边界决策、路线图）。本文只讲架构与 API。

## 三层架构（核心心智模型）

```
L3 编排层 (orchestrator/)   接单 → 动态生成 flow 代码 → 校验 → 执行（续跑锁定）
L2 引擎   (核心原语)        定义并跑好「单个 flow」：Checkpoint / 自改沙箱 / 质量门 / HITL / dry-run
L1 执行器 (agent.js+executor.js) 怎么驱动一个 CLI/agent + provider 能力分层 + 路由
```

- **L1 执行器**：`recursive`/`claude`/`cursor`/`gemini`/`codex`/`aider` 各有 adapter。
  执行器是否「接受外部 provider」由它有没有 `applyProvider` 翻译器派生——
  BYO-LLM（recursive/aider/claude）可注入 provider；锁定型（cursor 等）自管鉴权，给 provider 会 fail-fast。
- **L2 引擎**：原语都是可独立组合的一等公民，flow 只是它们的薄编排。
- **L3 codegen harness**：**codegen 为唯一主路径**（生成 flow 代码，与人手写同构），**不做 DAG**。
  三道护栏：约束式生成（词汇表+骨架）／跑前校验（语法+import 白名单+dry-run）／持久化+续跑锁定。

## 模块地图

| 文件 | 职责 |
|------|------|
| `index.js` | 公共 API 出口（所有对外导出都在这里登记） |
| `checkpoint.js` | `Checkpoint`：断点续跑的步骤记录（`cp.step` / `cp.done` / pauseContext） |
| `agent.js` | CLI adapter（claude/cursor/…/recursive）+ `runAgent` + `parallel`/`pipeline` + 可插拔 HITL（terminal/wecom）+ `recursiveProviderEnv` |
| `provider.js` | provider profile：`interpolateEnv`（`${VAR}`）+ `loadProviders` + `resolveProvider` + 通用 `loadMergedConfig` |
| `executor.js` | 执行器能力分层：`EXECUTORS` 注册表 + `getExecutor` + `loadAgents` + `resolveAgent`（绑定+校验） |
| `self-mod-guard.js` | `withSelfModGuard` / `captureBaseline`：自改安全沙箱（失败硬回滚） |
| `quality-gate.js` | `runGate` / `runGates`：质量门（rollback / resume-fix / autofix） |
| `failure-context.js` | `writeFailureContext` / `readAndConsumeFailureContext`：失败上下文（写入即消费，单轮热路径） |
| `memory.js` | `recordLearning` / `recall` / `buildMemorySection` / `promoteFailureContext`：轻量「跨-run」记忆（文件型 jsonl + 关键词/tag 召回，failure-context 的跨-run 超集） |
| `loop.js` | `loop`：goal-driven 循环原语（每轮 fresh context 迭代 + budget 封顶 + 复用 Checkpoint 续跑 + quality-gate 每轮硬验证 + 可选 memory 沉淀）。同步函数、跑完即返，不是 daemon |
| `git.js` | `gitStatus` / `gitDiff` / `gitCommitAll` / `gitWorktreeAdd` / `gitWorktreeRemove`：给生成的 flow 用的受控 git 原语（含 worktree 隔离） |
| `subflow.js` | `runFlow`：把另一条 flow 当隔离子进程跑；`fanOut`：并发编排多条子 flow（限并发 + worktree 隔离 + per-task 日志 + 汇总） |
| `dry-run.js` | `isDryRun`：dry-run 开关（`FLOWX_DRY_RUN`） |
| `orchestrator/` | L3：`FLOW_API.md`（契约）+ `templates/` + `examples/` + `validate.js` + `generate.js` + `decompose.js`（接单分拆）+ `run.js`（`orchestrate`/`orchestrateMulti`）+ `cli.js`（`flowx orchestrate [--split]`） |
| `dashboard/` | 可观测看板（只读）：`collect.js`（扫 `.flowx/runs` + worktree、重建父子树、僵尸推断）+ `render.js`（单文件 HTML）+ `cli.js`（`flowx dashboard [--open]`）。埋点经 `cp.event` / `setAgentEventSink` / 质量门 `onEvent` 写入 jsonl |
| `flows/` | 通用/示例 flow：`force-dev.js`、`goal-drive.js`。**项目特定 flow 放各自仓**（如 recursive 自改 flow 在 `recursive` 仓 `.dev/flows/`，ilink-hub 技术债消化 flow 在 `ilink-hub/.flowx/flows/`，均通过 `file:` 依赖消费本包） |
| `skills/flowx/` | flowx 配套 Claude Code skill：`SKILL.md`（主体，含路由表）+ `references/`（create/run/debug/config 四个参考文档）。安装：`cp -r skills/flowx ~/.claude/skills/` |
| `bin/flowx.js` | CLI 入口 |

## 常用命令

```bash
npm test                          # node --test test/（全量单测，含结构化 E2E，不烧 API）
node --check <file>               # 语法校验

flowx force-dev --feature x --repo .   # 跑 force-dev flow
flowx orchestrate "<目标>" --repo .     # L3：一行需求 → 生成 flow → 校验 → 执行（续跑锁定）
flowx orchestrate "<大目标>" --split    # L3 接单分拆：拆子任务 → 各自生成 flow → fanOut 并发执行
flowx dashboard --repo . [--open]      # 扫 .flowx/runs + worktree → 生成只读可观测看板 HTML
flowx run ./flows/my-flow.js [args]    # 跑自定义 flow
flowx list                             # 列出所有 run

# 自我迭代（recursive 仓自改）—— flow 在 recursive 仓，flowx 作为库被消费
# 见 recursive 仓 .dev/flows/SELF_IMPROVE.md
```

## 配置分层（重要）

provider / agent 配置**绝不硬编码在代码里**，按内容性质三分：

| 内容 | 放哪 |
|------|------|
| 通用 flow 逻辑 + adapter + provider/agent schema | **flowx 库**（本仓） |
| 项目特定 flow 配置（质量门、provider/agent 名） | **项目仓 `.flowx/`**（committed） |
| 机器级状态 + 密钥（run checkpoints、API key） | **`~/.flowx/`** 或 gitignore 的 `.flowx/` |

- `~/.flowx/providers.{json,yaml,js}`（机器级）→ `<repo>/.flowx/providers.*`（项目级覆盖）。
- 密钥用 `${ENV_VAR}` 运行时插值，**明文永不入仓**（缺变量 fail-fast）。
- 示例见 `examples/providers.example.json` / `examples/agents.example.json`。

## 开发约定

- **零运行时依赖**：核心不引第三方包（yaml 等为可选 lazy import）。新增依赖前先确认是否真的必要。
- **纯 ESM**：所有文件 `import`/`export`；新对外 API 必须在 `index.js` 登记。
- **原语优先**：先做可独立测试的一等公民原语，flow 只做薄编排。新原语必须带单测。
- **dry-run 可跑**：写 flow / 改原语时保证 `FLOWX_DRY_RUN=1` 下能跑通骨架（假执行器 + 假质量门）。
- **测试用假执行器**：E2E 用 fake CLI / fake gate / temp git repo，不烧 API、不依赖外部环境。
- **生成的 flow 受约束**：只能 import `flowcast`（+ `util`），只用 `FLOW_API.md` 列出的原语；
  需要 git/文件操作时走 flowx 暴露的原语，不裸调 `child_process`。

## Git 提交规范

提交时**不要**添加 `Co-Authored-By`、`Generated by` 等 AI 署名，直接用操作者的 git 身份提交。
